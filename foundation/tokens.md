# Токены

> Figma: TBD

Токены SEDA UI — единый язык между дизайном, кодом и документацией. Они описывают визуальные решения как именованные значения: цвет, типографику, отступы, скругления, границы, тени, фокус и параметры отдельных компонентов.

---

## Цель

Архитектура токенов нужна, чтобы:

- не использовать жёстко заданные визуальные значения в компонентах;
- поддерживать светлую и тёмную темы без переписывания компонентов;
- отделять базовую палитру от семантического назначения;
- управлять изменениями без хаотичных переименований;
- синхронизировать `tokens.json`, Figma variables, CSS variables и markdown-спецификации.

---

## Источник Правды

Текущий источник токенов: `tokens.json`.

Текущие коллекции:

| Коллекция | Режимы | Роль |
|---|---|---|
| `foundation` | `Mode 1` | Базовые токены: палитра, шрифты, отступы, границы |
| `semantic` | `Light`, `Dark` | Семантические токены с учётом темы |
| `tags` | `Gray`, `Blue`, `Green`, `Pistachio`, `Purple`, `Orange`, `Yellow`, `Red` | Цветовые токены тегов |
| `components` | `Mode 1` | Токены конкретных компонентов |

Markdown-документация должна описывать значения из `tokens.json`, а не вводить параллельную систему.

---

## Слои Токенов

### 1. Базовые Токены

Базовые токены — значения без продуктового смысла.

Примеры:

- `color/neutral/0`
- `color/blue/50`
- `font/body`
- `space/16`
- `border/radius/md`

Использовать:
- Только внутри семантических токенов, токенов компонентов или foundation-документов.
- Не использовать напрямую в спецификациях компонентов, если есть семантический эквивалент.

Не использовать:
- В production CSS компонента как основной контракт.
- В продуктовых спецификациях как ответ на вопрос “какой смысл у цвета?”.

### 2. Семантические Токены

Семантические токены описывают назначение, а не конкретное значение.

Примеры:

- `surface/default`
- `container/brand/default`
- `border/hover`
- `text/primary`
- `status/error/text`
- `focus/ring`

Использовать:
- В токенах компонентов.
- В спецификациях как основной публичный смысл.
- Для светлой и тёмной темы.

Не использовать:
- Для уникальных деталей одного компонента, если нужен токен компонента.
- Для состояния, которого нет в `state-vocabulary.md`, без явного обоснования.

### 3. Токены Компонентов

Токены компонентов — локальные токены компонента. Они связывают конкретную часть компонента с семантическими токенами.

Примеры:

- `--btn-primary-bg`
- `--input-border-focus`
- `--modal-overlay`
- `--tabs-indicator`

Использовать:
- В CSS/реализации компонентов.
- В спецификациях компонентов.
- Для настройки конкретного компонента.

Правило:
- Токен компонента должен ссылаться на семантический токен, базовый токен или задокументированное исключение.
- Для кастомизации компонента переопределяйте токен компонента, а не семантический токен.

### 4. Токены Режимов

Токены режимов — значения токенов в конкретном режиме.

Текущие режимы:

- `semantic/Light`
- `semantic/Dark`
- цветовые режимы тегов: `Gray`, `Blue`, `Green`, `Pistachio`, `Purple`, `Orange`, `Yellow`, `Red`

Использовать:
- Для тем и цветовых вариантов.
- Для платформенных и тематических адаптеров.

Не использовать:
- Как часть API компонента.

## Правила Именования

SEDA использует slash naming для design tokens и CSS custom properties для implementation tokens.

### Дизайн-Токены

Общий шаблон:

```text
category / role / variant / state
```

Не все уровни обязательны. Используйте минимальное имя, которое остается однозначным.

| Уровень | Пример | Значение |
|---|---|---|
| `category` | `surface`, `text`, `border` | Тип визуального свойства |
| `role` | `brand`, `inverse`, `error` | Семантическая роль |
| `variant` | `primary`, `secondary`, `container` | Уточнение внутри роли |
| `state` | `default`, `hover`, `pressed`, `disabled` | Состояние |

Примеры:

- `surface/default`
- `surface/brand/hover`
- `text/on-brand/primary`
- `status/error/border`
- `link/visited-pressed`

### Токены Компонентов

CSS custom properties используют kebab-case:

```text
--component-part-role-state
```

Примеры:

- `--btn-primary-bg-hover`
- `--input-border-error`
- `--modal-header-border`
- `--tag-focus-ring`

Правила:

- Используйте короткий component prefix: `btn`, `input`, `modal`, `tabs`.
- Суффикс состояния должен совпадать со словарем состояний: `hover`, `pressed`, `focus`, `selected`, `checked`, `disabled`, `loading`.
- Не смешивайте `active` и `selected`: `active` — момент нажатия, `selected` — устойчивый выбор.

## Правила Категорий

### `surface`

Использовать для:
- фона страницы;
- фона секции;
- базовых неинтерактивных поверхностей;
- инверсных и брендовых поверхностей.

Не использовать для:
- заливок кнопок и контролов, если есть `container`;
- статусных сообщений, если есть `status/*`.

### `container`

Использовать для:
- заливок интерактивных контролов;
- фонов кнопок;
- выбираемых областей;
- заливок hover/pressed у контролов.

Не использовать для:
- фона всей страницы;
- цвета текста;
- границ.

### `border`

Использовать для:
- обводок компонентов;
- разделителей;
- границ контролов;
- изменений границы рядом с фокусом.

Не использовать для:
- самого focus ring, используйте `focus/ring`;
- теней или elevation.

### `text`

Использовать для:
- основного текста;
- подписей;
- иконок, если они используют общий foreground с текстом;
- текста на brand/inverse поверхностях.

Не использовать для:
- границ или фона;
- статусного текста, если есть токен `status/*/text`.

### `link`

Использовать для:
- навигационных текстовых ссылок;
- посещённых ссылок;
- inline-ссылок в текстовом контенте.

Не использовать для:
- кнопок;
- табов;
- пунктов меню, которые запускают действия.

### `status`

Использовать для:
- состояний валидации;
- статусов Alert/Toast;
- семантической обратной связи: info, error, warning, success, disabled.

Не использовать для:
- чисто декоративной цветовой кодировки;
- брендового акцента;
- произвольных цветов тегов.

### `focus`

Использовать для:
- индикатора клавиатурного фокуса.

Не использовать для:
- selected/current состояния;
- hover или active.

### `shadow`

Использовать для:
- реализационных значений тени внутри уровней, описанных в `elevation.md`;
- компонентов, которые перекрывают или визуально поднимаются над контентом.

Примечание о текущем состоянии:
- `shadow/base` и `shadow/darker` сейчас описаны в `theming.md`.
- Решение о глубине принимает `elevation.md`; shadow-токены остаются значениями реализации.

### `tag`

Использовать для:
- семантических цветовых вариантов тегов.

Не использовать для:
- общих статусных сообщений;
- кнопок или ссылок.

## Именование Состояний

Названия состояний должны соответствовать `state-vocabulary.md`.

| Состояние | Значение | Суффикс токена |
|---|---|---|
| `default` | Базовое состояние | `default` |
| `hover` | Наведение указателя | `hover` |
| `active` | Момент нажатия | В визуальных токенах предпочтительно `pressed` |
| `focus` | Клавиатурный фокус | `focus` |
| `selected` | Устойчивый выбор одного варианта | `selected` |
| `checked` | Бинарная отметка | `checked` |
| `disabled` | Недоступно | `disabled` |
| `loading` | Асинхронная операция | `loading` |
| `error` | Ошибка или невалидное состояние | `error` |
| `warning` | Потенциальная проблема | `warning` |
| `success` | Успешное состояние | `success` |

Правило:
- Используйте `pressed` для визуальных токенов, которые описывают вид в момент нажатия.
- Используйте `active` только при описании самого interaction state.

## Ссылки Между Токенами

Токены могут ссылаться на другие токены.

Разрешено:

```text
component token -> semantic token
semantic token -> primitive token
semantic token -> semantic token
```

Избегать:

```text
component token -> component token from another component
primitive token -> semantic token
primitive token -> component token
```

Задокументированные исключения:

- `transparent` можно использовать там, где визуальный слой должен отсутствовать.
- raw `rgba(...)` можно использовать для focus или overlay только до появления эквивалентных токенов.

---

## Примечания О Текущем Состоянии

Известные расхождения документации и токенов, которые нужно убрать при работе над token pipeline:

- Некоторые specs ссылаются на `status/error/container/default`, `status/error/container/hover`, `status/error/container/pressed`, хотя текущие semantic tokens используют `status/error/container` без суффиксов состояния.
- Некоторые specs ссылаются на `border-radius-lg`; новые specs должны использовать `border/radius/*` из `radius-border.md`.
- `shadow/base` и `shadow/darker` используются как semantic tokens; уровни глубины описаны в `elevation.md`.
- `transparent` встречается в таблицах component tokens как semantic value; оставляем это как задокументированное исключение, пока token linting не поддерживает явные null/transparent значения.
- `tokens.json` содержит более богатые данные по шрифтам, чем старые markdown-документы. Теперь `typography.md` является foundation-источником для типографических ролей.

## Жизненный Цикл Токена

### Добавление

Добавляйте токен только если:

- существующие токены не выражают нужный смысл;
- у токена есть хотя бы один известный сценарий в компоненте или foundation;
- для semantic token определено поведение в Light/Dark;
- роль токена задокументирована.

### Переименование

Переименовывайте только если:

- текущее имя вводит в заблуждение;
- новое имя улучшает семантическую ясность;
- можно описать миграцию.

Переименования являются breaking changes для реализации компонентов.

### Устаревание

Перед удалением:

1. Отметьте токен как deprecated в документации.
2. Укажите токен-замену.
3. Обновите specs компонентов.
4. Сохраните совместимость минимум на один release cycle.

### Удаление

Удаляйте только после того, как:

- на токен не ссылается ни один component spec;
- реализация больше его не использует;
- changelog содержит заметки о миграции.

## Требования К Документации

Каждая задокументированная группа токенов должна включать:

- имя токена;
- роль;
- значение Light/Dark или поведение по режимам;
- разрешённое использование;
- ограничения, если токен легко применить неправильно;
- accessibility note, если токен влияет на контраст или фокус.

Таблицы component tokens должны включать:

| Колонка | Значение |
|---|---|
| `Component token` | CSS custom property или токен реализации |
| `Role` | Какая часть компонента управляется токеном |
| `Semantic token` | Семантический токен, на который идёт ссылка |
| `Notes` | Исключения или поведение состояния |

---

## Проверки Качества

Проверки токенов должны находить:

- semantic token без Light или Dark значения;
- component token со ссылкой на неизвестный semantic token;
- токен из spec, которого нет в `tokens.json`;
- неиспользуемый токен;
- дубликат с тем же значением и той же ролью;
- raw hex/rgba в component specs без задокументированного исключения;
- суффикс состояния, которого нет в `state-vocabulary.md`;
- placeholder token names в опубликованных specs.

---

## Что Делать И Чего Избегать

| Делать | Избегать |
|---|---|
| Использовать semantic tokens в component specs | Использовать primitive colors напрямую в component specs |
| Использовать component tokens для локальной настройки | Переопределять semantic tokens ради одного компонента |
| Сверять имена состояний с `state-vocabulary.md` | Использовать `active` в значении selected |
| Документировать разрешённое использование | Добавлять имена токенов без роли |
| Помечать deprecated перед удалением | Переименовывать токены молча |
| Проверять Light и Dark вместе | Добавлять Light-only semantic tokens |

---

## Чеклист Предложения Токена

- [ ] Слой токена понятен: primitive, semantic, component или mode.
- [ ] Имя соответствует slash naming или правилам CSS custom property.
- [ ] Роль задокументирована.
- [ ] Разрешённое использование задокументировано.
- [ ] Для semantic token определены Light/Dark значения.
- [ ] Проверено влияние на accessibility.
- [ ] Существующий токен не решает ту же задачу.
- [ ] Component specs, которые ссылаются на токен, обновлены.
- [ ] Есть migration note, если токен заменяет старый.

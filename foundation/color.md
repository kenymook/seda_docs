# Цвет

> Figma: TBD

Color foundation описывает, как SEDA UI использует цвет для иерархии, состояний, доступности и тематизации. Конкретные значения semantic tokens для Light/Dark перечислены в `theming.md`; архитектура слоёв токенов описана в `tokens.md`.

---

## Цель

Цвет в SEDA UI должен:

- поддерживать читаемость и сканирование интерфейса;
- отделять фон, контейнеры, границы и foreground;
- ясно передавать интерактивные, валидационные и статусные состояния;
- работать в Light и Dark темах;
- соответствовать критериям доступности;
- не быть единственным способом передать важный смысл.

SEDA UI — система для плотных B2B и AI-интерфейсов, поэтому цвет должен быть сдержанным и функциональным. Данные, действия и состояния важнее декоративной насыщенности.

---

## Источники Токенов

| Слой | Примеры | Назначение |
|---|---|---|
| Базовая палитра | `color/neutral/0`, `color/blue/50`, `color/red/50` | Исходные цветовые значения |
| Alpha-палитра | `color/black/5`, `color/white/10` | Overlay, мягкие заливки, границы |
| Семантический цвет | `surface/default`, `text/primary`, `status/error/text` | Смысл и поведение в теме |
| Цвет компонента | `--btn-primary-bg`, `--input-border-error` | Использование внутри конкретного компонента |
| Цветовые режимы тегов | `Gray`, `Blue`, `Green`, `Pistachio`, `Purple`, `Orange`, `Yellow`, `Red` | Варианты тегов |

Правило: спецификации компонентов должны использовать semantic или component tokens, а не primitive colors напрямую.

---

## Базовая Палитра

Текущие базовые группы в `tokens.json`:

| Группа | Шаги | Роль |
|---|---|---|
| `neutral` | `0`-`100` | Базовые фоны, foreground, grayscale UI |
| `black` | alpha steps `5`-`95` | Мягкие границы/заливки на светлых поверхностях |
| `white` | alpha steps `5`-`95` | Мягкие границы/заливки на тёмных поверхностях |
| `blue` | `5`-`95` | Brand, info, links, акценты, связанные с фокусом |
| `red` | `5`-`95` | Ошибка, destructive states |
| `yellow` | `5`-`95` | Warning states |
| `green` | `5`-`95` | Success states |
| `purple` | `5`-`95` | Посещённые ссылки и опциональный категорийный цвет |
| `orange` | `5`-`95` | Опциональный категорийный или status-adjacent цвет |
| `pistachio` | `5`-`95` | Опциональный цвет тегов/категорий |
| `transparent` | одно значение | Явное отсутствие заливки |

Primitive tokens не задают смысл. Перед использованием в компонентах всегда проводите их через semantic roles.

---

## Семантические Цветовые Роли

### Поверхность

Используйте `surface/*` для широких, в основном неинтерактивных фонов:

- фон страницы;
- фон секции;
- фон карточки или body-области;
- inverse/brand поверхность.

| Семейство токенов | Использование |
|---|---|
| `surface/default` | Базовая поверхность страницы |
| `surface/hover` | Hover-фон для мягких не-control поверхностей |
| `surface/pressed` | Pressed-фон для мягких поверхностей |
| `surface/inverse/*` | Инверсные секции и dark-on-light сценарии |
| `surface/brand/*` | Брендовые поверхности, selected/brand emphasis |

Не используйте `surface` для заливок кнопок, если `container` лучше выражает семантику.

### Контейнер

Используйте `container/*` для заливок контролов и интерактивных областей:

- кнопки;
- segmented controls;
- выбираемые строки;
- интерактивные chips/tags;
- контролы тулбара.

| Семейство токенов | Использование |
|---|---|
| `container/default` | Нейтральная заливка контрола |
| `container/hover` | Заливка контрола при hover |
| `container/pressed` | Заливка контрола при pressed |
| `container/brand/*` | Заливка primary/selected/checked контрола |
| `container/inverse/*` | Контролы на inverse surfaces |

### Граница

Используйте `border/*` для видимых границ:

- границы input;
- outlines карточек;
- разделители;
- разделители строк таблицы;
- outlines компонентов.

| Семейство токенов | Использование |
|---|---|
| `border/default` | Базовая мягкая граница |
| `border/hover` | Более заметная граница при hover |
| `border/pressed` | Граница при pressed |
| `border/brand/*` | Brand emphasis или граница рядом с focus |
| `border/inverse/*` | Границы на inverse surfaces |
| `border/negative/*` | Legacy-уровни negative/error границ |

Примечание о текущем состоянии: валидационные границы должны предпочитать `status/error/border`, `status/warning/border`, `status/success/border`. `border/negative/*` нужно пересмотреть во время чистки токенов.

### Текст

Используйте `text/*` для foreground-контента:

- основной текст;
- labels;
- helper text;
- иконки, которые наследуют цвет текста;
- foreground на brand/inverse surfaces.

| Токен | Использование |
|---|---|
| `text/primary` | Основной читаемый текст |
| `text/secondary` | Вспомогательный текст |
| `text/tertiary` | Подписи, метаданные, low-emphasis text |
| `text/muted` | Placeholder и приглушённые подсказки |
| `text/on-brand/*` | Текст на brand backgrounds |
| `text/on-inverse/*` | Текст на inverse backgrounds |

Не используйте только opacity для понижения акцента, если существует semantic text token.

### Ссылка

Используйте `link/*` только для навигации к ресурсу или маршруту.

| Токен | Использование |
|---|---|
| `link/default` | Ссылка по умолчанию |
| `link/hover` | Ссылка при hover |
| `link/pressed` | Ссылка при pressed |
| `link/visited` | Посещённая ссылка |
| `link/visited-hover` | Посещённая ссылка при hover |
| `link/visited-pressed` | Посещённая ссылка при pressed |

Ссылки в тексте должны различаться не только цветом: для inline-ссылок нужно подчёркивание или другой постоянный индикатор.

### Статус

Используйте `status/*` для семантической обратной связи:

- информация;
- ошибка;
- предупреждение;
- успех;
- отключённое состояние.

| Семейство | Использование |
|---|---|
| `status/info/*` | Нейтральная информационная обратная связь |
| `status/error/*` | Ошибка, destructive, invalid |
| `status/warning/*` | Предупреждение, риск, caution |
| `status/success/*` | Успех, valid, completed |
| `status/disabled/*` | Disabled surface/container/border/text |

Status color должен сопровождаться текстом, иконкой или структурным сигналом, если он передаёт важный смысл.

### Тег

Используйте цвета тегов для категорийных меток и фильтрующих метаданных. Не используйте цвета тегов как общие status colors, если содержание тега не категорийное, а системное.

---

## Цветовые Режимы

SEDA сейчас поддерживает:

- `Light`;
- `Dark`;
- цветовые режимы тегов.

Light/Dark меняет значения semantic tokens, а не структуру компонента.

Правила:

- Компонент ссылается на один и тот же semantic token в любой теме.
- Semantic token определяет значения для конкретных режимов.
- Спецификации компонентов не должны вручную выбирать разные primitive colors для Light/Dark.
- Если у semantic token нет Dark-значения, он не готов для production.

---

## Модель Поверхностей И Слоёв

Текущие semantic tokens включают `surface/default`, `container/default`, `shadow/base` и component tokens для overlay. Правила визуальной глубины описаны в `elevation.md`; ниже указана рекомендуемая модель будущих layer tokens.

Рекомендуемая модель слоёв для следующей итерации токенов:

| Будущий токен | Роль |
|---|---|
| `layer/01` | Базовый фон страницы |
| `layer/02` | Карточки, панели, контейнеры таблиц |
| `layer/03` | Приподнятые панели, sticky bars |
| `layer/overlay` | Dropdown, popover, floating surfaces |
| `overlay/scrim` | Backdrop модального окна или drawer |
| `surface/raised` | Поверхность с elevation |

Пока этих layer-токенов нет:

- используйте `surface/default` для базовых поверхностей;
- используйте component tokens для overlays: `--modal-bg`, `--popover-bg`, `--drawer-bg`;
- выбирайте уровень глубины по `elevation.md`;
- используйте `shadow/base` или `shadow/darker` как реализационное значение выбранного уровня.

---

## Текст На Цвете

Используйте парные foreground tokens для цветных поверхностей.

| Фон | Foreground |
|---|---|
| `surface/brand/default` | `text/on-brand/primary` |
| `container/brand/default` | `text/on-brand/primary` |
| `surface/inverse/default` | `text/on-inverse/primary` |
| Сильная status surface | status-specific text или `text/on-brand/primary`, только если проходит контраст |
| Базовая поверхность | `text/primary`, `text/secondary`, `text/tertiary`, `text/muted` |

Правила:

- Не размещайте `text/primary` на brand surface без проверки контраста.
- Не размещайте `text/on-brand/*` на default surface.
- Muted text на цветном фоне всё равно должен соответствовать требованиям доступности, если он несёт смысл.

---

## Требования К Контрасту

SEDA следует `foundation/accessibility.md`.

| Пара | Минимум |
|---|---:|
| Основной текст / фон | `4.5:1` |
| Label/helper text / фон | `4.5:1` |
| Крупный текст / фон | `3:1` |
| Иконка / соседний цвет | `3:1` |
| Интерактивная граница / соседний цвет | `3:1` |
| Focus indicator / соседние цвета | `3:1` |

Disabled content может иметь более низкий контраст только если он действительно недоступен и не нужен для выполнения задачи.

---

## Передача Состояний

Цвет может поддерживать состояние, но не должен быть единственным сигналом.

| Состояние | Обязательная нецветовая поддержка |
|---|---|
| `error` | Текст ошибки, иконка или связь с полем |
| `warning` | Текст предупреждения или иконка |
| `success` | Текст успеха, иконка или сообщение о завершении |
| `selected` | Форма, check, граница, индикатор или `aria-selected` |
| `current` | `aria-current`, индикатор или текстовое выделение |
| `destructive` | Явный label и обычно подтверждение |
| `disabled` | Поведение disabled, а не только приглушённый цвет |

## Использование Статусов

### Информация

Использовать для:
- нейтральных объявлений;
- контекстных подсказок;
- нейтральных сообщений о прогрессе.

Не использовать для:
- брендового продвижения;
- selected state, если `container/brand/*` понятнее.

### Ошибка

Использовать для:
- невалидного ввода;
- неудачной операции;
- подтверждение destructive-действия;
- блокирующей проблемы.

Правила:
- Всегда сопровождать поясняющим текстом.
- Текст ошибки должен говорить, как исправить ситуацию.
- Не показывайте ошибку на пустых untouched fields по умолчанию.

### Предупреждение

Использовать для:
- потенциальной проблемы;
- рискованного, но разрешённого действия;
- приближения к лимиту.

Правила:
- Warning не всегда блокирует действие.
- Объясняйте последствие или следующий шаг.

### Успех

Использовать для:
- завершённой операции;
- valid state;
- положительного подтверждения.

Правила:
- Избегайте постоянных success states, которые добавляют шум.
- Не используйте success color как декоративный зелёный.

### Отключённое Состояние

Использовать для:
- недоступных контролов;
- disabled text/border/container.

Правила:
- Disabled state должен также отключать взаимодействие.
- Объясняйте причину, если пользователю нужно понять, как разблокировать действие.

---

## Бренд И Акцент

`blue` сейчас является основной цветовой группой для бренда и действий.

Используйте brand color для:

- primary actions;
- selected/checked controls;
- links;
- акценты рядом с focus, когда это задокументировано;
- моментов продуктовой идентичности.

Не используйте brand color для:

- каждой иконки на плотном экране;
- декоративных фонов за данными;
- статусного смысла, когда `status/*` точнее.

---

## Цвета Тегов

Цвета тегов категорийные, а не обязательно семантическая обратная связь.

Использовать для:
- категорий;
- меток;
- фильтров;
- таксономии;
- пользовательской группировки.

Не использовать для:
- валидации форм;
- alerts;
- destructive actions;
- progress states.

Если тег означает системный статус, предпочтительнее использовать явный status component или status token.

---

## Примечание О Визуализации Данных

SEDA пока не определяет цветовые токены для графиков и data-viz.

Пока data-viz tokens не существуют:

- не используйте status colors для произвольных рядов графика;
- не полагайтесь только на hue для различения рядов;
- добавляйте подписи, легенды, паттерны или прямые аннотации;
- по возможности сохраняйте red/yellow/green для реального семантического статуса.

Рекомендуемая будущая группа токенов:

- `data/series/01`;
- `data/series/02`;
- `data/diverging/negative`;
- `data/diverging/positive`;
- `data/sequential/*`.

## Что Делать И Чего Избегать

| Делать | Избегать |
|---|---|
| Использовать semantic color tokens в specs компонентов | Использовать primitive palette напрямую в CSS компонента |
| Сочетать цветные поверхности с подходящими foreground tokens | Слепо ставить `text/primary` на brand backgrounds |
| Использовать status colors только для семантической обратной связи | Использовать red/yellow/green как декор |
| Делать inline-ссылки различимыми не только цветом | Полагаться только на синий текст в body copy |
| Проверять Light и Dark вместе | Выпускать Light-only цветовые решения |
| Использовать tag colors для категорий | Использовать tag colors как validation states |

---

## Чеклист Проверки Цвета

- [ ] Компонент использует semantic или component tokens, а не raw hex.
- [ ] Light и Dark режимы оба определены.
- [ ] Контраст текста проходит требуемый ratio.
- [ ] Иконки и интерактивные границы проходят `3:1`.
- [ ] Focus indicator остаётся видимым.
- [ ] Статус не передаётся только цветом.
- [ ] Brand color не используется там, где status color точнее.
- [ ] Tag color не используется как validation/status feedback.
- [ ] Disabled state меняет поведение, а не только цвет.
- [ ] Любое использование raw `rgba` или `transparent` задокументировано как исключение.

# Content Design И UX Writing

> Figma: TBD

Content design описывает, как SEDA UI пишет интерфейсные тексты: labels, кнопки, ошибки, подсказки, empty states, loading states, подтверждения и системные сообщения. Текст является частью UX и accessibility, а не декоративным слоем.

---

## Цель

Тексты в SEDA UI должны:

- помогать пользователю понять, что происходит;
- ясно называть действия и последствия;
- снижать когнитивную нагрузку в плотных B2B и AI-интерфейсах;
- быть короткими, конкретными и полезными;
- работать вместе с accessibility, typography и validation rules;
- не перекладывать смысл только на цвет, иконку или состояние.

Правило: хороший интерфейсный текст отвечает на один из вопросов пользователя: что это, что случилось, что делать дальше или что изменится после действия.

---

## Voice And Tone

SEDA UI говорит спокойно, ясно и по делу.

| Качество | Что означает |
|---|---|
| Clear | Текст понятен без знания внутренней терминологии команды |
| Concise | Нет лишних слов и объяснений очевидного |
| Useful | Текст помогает принять решение или исправить ситуацию |
| Calm | Ошибки и предупреждения не звучат драматично |
| Action-oriented | Действия названы глаголом и ведут к ожидаемому результату |

Тон может быть более строгим в destructive и security-сценариях, но не должен становиться пугающим или обвиняющим.

---

## Общие Правила

- Пишите для пользователя, а не для структуры базы данных.
- Используйте конкретные существительные: `Проект`, `Пользователь`, `Отчёт`, `Интеграция`.
- Используйте глаголы для действий: `Сохранить`, `Удалить`, `Отправить`, `Создать проект`.
- Не используйте `OK`, `Да`, `Нет`, если можно назвать действие точнее.
- Не начинайте сообщение с технической причины, если пользователю важен следующий шаг.
- Не дублируйте один и тот же смысл в title, description и button label.
- Не используйте ALL CAPS для обычных UI labels.

---

## Термины И Капитализация

| Контекст | Правило | Пример |
|---|---|---|
| Button label | Глагол или глагольная фраза | `Сохранить`, `Создать проект` |
| Field label | Короткое существительное | `Email`, `Название проекта` |
| Menu item | Действие или объект | `Переименовать`, `Экспорт` |
| Tab label | Название раздела | `Настройки`, `История` |
| Table header | Короткая подпись колонки | `Статус`, `Владелец` |
| Empty state title | Состояние или причина | `Проектов пока нет` |
| Error text | Проблема + способ исправить | `Введите email в формате name@example.com` |

Используйте sentence case для русского интерфейса: первое слово с заглавной буквы, остальные по правилам языка.

## Button Labels

Button label должен называть действие, которое произойдёт после нажатия.

| Сценарий | Хорошо | Избегать |
|---|---|---|
| Сохранение формы | `Сохранить` | `OK` |
| Создание объекта | `Создать проект` | `Создать` без контекста, если рядом несколько объектов |
| Отправка | `Отправить заявку` | `Далее`, если действие уже финальное |
| Отмена | `Отменить` | `Нет` |
| Закрытие | `Закрыть` | `Понятно`, если это не подтверждение понимания |
| Удаление | `Удалить проект` | `Да`, `Подтвердить` |

Правила:

- Один экран или диалог обычно имеет одно главное действие.
- Destructive action явно называет объект: `Удалить проект`, `Отозвать доступ`.
- Loading state сохраняет смысл действия. Spinner не заменяет accessible name.
- Icon-only button требует `aria-label` и Tooltip с тем же смыслом.

---

## Links

Link ведёт к ресурсу, маршруту или дополнительной информации. Button запускает действие.

| Сценарий | Хорошо | Избегать |
|---|---|---|
| Документация | `Открыть документацию` | `Здесь` |
| Подробности | `Подробнее о лимитах` | `Подробнее`, если ссылок несколько |
| Внешний ресурс | `Открыть в Stripe` | `Перейти` без контекста |

Inline link должен быть понятен вне окружающего предложения, если screen reader читает список ссылок.

## Field Labels, Placeholders И Helper Text

### Field Label

Label всегда видим и программно связан с полем.

| Хорошо | Избегать |
|---|---|
| `Email` | Placeholder вместо label |
| `Название проекта` | `Введите название проекта` как label |
| `Дата начала` | `Дата` для пары start/end без уточнения |

Label отвечает на вопрос “что это за поле?”, а не “что нужно сделать?”.

### Placeholder

Placeholder показывает пример формата, но не заменяет label.

| Хорошо | Избегать |
|---|---|
| `name@example.com` | `Email`, если label уже `Email` |
| `https://example.com` | Длинная инструкция |
| `Иван Петров` | Обязательная информация, которая исчезает при вводе |

### Helper Text

Helper text объясняет формат, ограничение или последствие.

| Сценарий | Пример |
|---|---|
| Формат | `Используйте рабочий email` |
| Ограничение | `До 120 символов` |
| Последствие | `Это имя увидят участники команды` |
| Безопасность | `Пароль должен содержать минимум 12 символов` |

Helper text не должен повторять label или placeholder.

## Validation Messages

Validation message должен объяснять проблему и, когда возможно, способ исправления.

| Тип | Цель | Пример |
|---|---|---|
| `error` | Блокирует действие, пока проблема не исправлена | `Введите email в формате name@example.com` |
| `warning` | Предупреждает о риске, но не всегда блокирует | `Вы близки к лимиту проектов` |
| `success` | Подтверждает результат, если это действительно полезно | `Изменения сохранены` |

Правила:

- Не показывайте ошибку на пустом untouched field.
- Показывайте ошибку после blur, изменения поля или попытки отправки формы.
- Не пишите “Неверное значение” без объяснения.
- Не обвиняйте пользователя: `Не удалось сохранить` лучше, чем `Вы ввели неправильно`.
- Success не заменяет label или helper text.
- Warning должен объяснять последствие или следующий шаг.

### Error Patterns

| Сценарий | Хорошо | Избегать |
|---|---|---|
| Required field | `Введите название проекта` | `Обязательное поле` |
| Wrong format | `Введите email в формате name@example.com` | `Неверный email` |
| Too long | `Сократите текст до 120 символов` | `Превышен лимит` |
| Network | `Не удалось сохранить. Проверьте соединение и попробуйте снова` | `Ошибка 500` |
| Permission | `У вас нет доступа к этому проекту` | `Forbidden` |

Технические коды можно показывать в detail/debug context, но не как основное сообщение.

## Empty States

Empty State объясняет отсутствие данных и предлагает следующий шаг.

| Тип | Title | Description | Action |
|---|---|---|---|
| `no-data` | `Проектов пока нет` | `Создайте первый проект, чтобы начать работу` | `Создать проект` |
| `no-results` | `Ничего не найдено` | `Измените запрос или сбросьте фильтры` | `Сбросить фильтры` |
| `no-access` | `Нет доступа` | `Запросите доступ у владельца проекта` | `Запросить доступ` |
| `error` | `Не удалось загрузить данные` | `Проверьте соединение и попробуйте снова` | `Повторить` |
| `first-time` | `Настройте рабочее пространство` | `Добавьте команду и подключите первые источники данных` | `Начать настройку` |

Правила:

- Не используйте только `Нет данных`.
- Title кратко называет состояние.
- Description объясняет причину или следующий шаг.
- Action нужен, если пользователь может что-то сделать прямо сейчас.
- Если действия нет, объясните почему.

## Loading И Async Text

Loading text должен подтверждать, что система работает, но не перегружать интерфейс.

| Сценарий | Текст |
|---|---|
| Button loading | Сохраняйте accessible name действия: `Сохранить` |
| Full section loading | `Загрузка...` или Skeleton без видимого текста, если есть `aria-busy` |
| Known progress | `Загружено 45 %` |
| Long operation | `Импорт может занять несколько минут` |
| Completed | `Импорт завершён` |
| Failed | `Импорт не завершён. Проверьте файл и попробуйте снова` |

Не показывайте loading text для очень коротких операций, если это создаёт flicker.

## Confirmation И Destructive Actions

Destructive action требует явного текста, который называет действие, объект и последствие.

| Часть | Пример |
|---|---|
| Title | `Удалить проект?` |
| Description | `Проект и все связанные данные будут удалены без возможности восстановления.` |
| Primary destructive action | `Удалить проект` |
| Safe action | `Отменить` |

Правила:

- Не используйте `Да` / `Нет`.
- Не ставьте фокус по умолчанию на destructive action.
- Не скрывайте последствие в Tooltip.
- Для irreversible actions используйте Modal confirmation.
- Если действие обратимо, укажите это: `Вы сможете восстановить проект в течение 30 дней`.

---

## Tooltips

Tooltip дополняет понятный control, но не заменяет label.

Использовать для:

- icon-only controls;
- короткого пояснения необычного действия;
- уточнения сокращённого значения;
- объяснения disabled причины, если она не очевидна.

Не использовать для:

- обязательной инструкции;
- длинной документации;
- ошибки формы;
- подтверждения destructive action.

Tooltip text должен быть коротким: обычно 2-6 слов.

## Notifications: Alert И Toast

### Alert

Alert остаётся в потоке страницы и используется для контекстной информации.

| Тип | Текст |
|---|---|
| `info` | Нейтральное сообщение или рекомендация |
| `success` | Подтверждение завершения, если оно полезно в контексте |
| `warning` | Риск или ограничение |
| `error` | Проблема, которую нужно исправить |

### Toast

Toast — короткое уведомление поверх интерфейса.

| Сценарий | Пример |
|---|---|
| Success | `Изменения сохранены` |
| Copy | `Ссылка скопирована` |
| Undo | `Проект удалён` + action `Отменить` |
| Error | `Не удалось сохранить` |

Toast не должен содержать важную информацию, которая исчезнет до того, как пользователь сможет её использовать.

## AI И Автоматизация

Для AI-интерфейсов тексты должны быть особенно явными:

- различайте “система думает”, “система выполняет действие” и “нужен ввод пользователя”;
- показывайте, что можно отменить или отредактировать;
- не обещайте абсолютную точность, если результат вероятностный;
- используйте формулировки вроде `Черновик готов`, `Проверьте результат перед отправкой`;
- не скрывайте важные ограничения модели или источника данных.

Примеры:

| Сценарий | Хорошо | Избегать |
|---|---|---|
| Генерация | `Готовим черновик ответа` | `Создаём идеальный ответ` |
| Результат | `Черновик готов` | `Готово`, если нужен review |
| Ошибка | `Не удалось получить ответ. Попробуйте снова` | `AI failed` |
| Ограничение | `Ответ основан на доступных данных` | `Ответ точный` |

## Accessibility

- Видимый label предпочтительнее скрытого `aria-label`.
- `aria-label` для icon-only control должен описывать действие, а не иконку: `Удалить`, не `Крестик`.
- Error text должен быть связан с полем через `aria-describedby`.
- Link text должен быть понятен без окружающего контекста.
- Live region должен сообщать изменение кратко и не перечитывать весь экран.
- Не используйте только цвет или иконку для важных статусов.

---

## Что Делать И Чего Избегать

| Делать | Избегать |
|---|---|
| Называть действие глаголом | Использовать `OK`, `Да`, `Нет` без контекста |
| Объяснять, как исправить ошибку | Писать `Ошибка` или `Неверное значение` |
| Держать label видимым | Заменять label placeholder |
| Давать empty state следующий шаг | Показывать тупиковое `Нет данных` |
| Называть destructive объект | Писать `Удалить` без объекта в опасном сценарии |
| Писать коротко и конкретно | Объяснять очевидное длинными предложениями |

## Чеклист

- [ ] Button labels используют глагол или глагольную фразу.
- [ ] Destructive actions называют объект и последствие.
- [ ] Field labels видимы и не заменены placeholder.
- [ ] Placeholder показывает пример, а не обязательную инструкцию.
- [ ] Helper text не дублирует label.
- [ ] Error объясняет, что исправить.
- [ ] Empty state объясняет причину и следующий шаг.
- [ ] Loading state не скрывает accessible name действия.
- [ ] Toast не содержит критичную информацию, которая исчезнет.
- [ ] Icon-only controls имеют `aria-label` и Tooltip с действием.

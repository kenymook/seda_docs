# SEDA AI Maturity Model

Модель зрелости SEDA AI помогает оценить, насколько дизайн-система готова развиваться как AI-ready design system framework и безопасно поддерживать AI-assisted product development.

## Уровни зрелости

| Уровень | Название | Краткое описание |
| --- | --- | --- |
| Level 0 | Interface Chaos | Интерфейсы создаются разрозненно, правила не зафиксированы |
| Level 1 | Basic Design System | Есть базовые элементы и визуальные правила, но система неполная |
| Level 2 | Systematic Design System | Токены, компоненты и документация связаны в рабочую систему |
| Level 3 | AI-ready Design System | Система описана так, чтобы AI мог безопасно помогать внутри правил |
| Level 4 | AI-assisted Interface Infrastructure | AI встроен в процессы проектирования, Handoff, Validation и Governance |

## Сравнение уровней

| Критерий | Level 0 | Level 1 | Level 2 | Level 3 | Level 4 |
| --- | --- | --- | --- | --- | --- |
| Design Tokens | Отсутствуют или случайны | Есть базовые цвета и типографика | Есть семантическая архитектура | Токены проверяемы и AI-readable | Токены связаны с автоматической Validation |
| Components | Разрозненные элементы | Библиотека основных компонентов | Компоненты имеют варианты и состояния | Компоненты имеют specs, contracts и AI rules | Компоненты интегрированы в AI-assisted workflows |
| Documentation | Почти отсутствует | Описывает внешний вид | Описывает применение и ограничения | Служит source of truth для людей и AI | Используется в автоматизированном контроле качества |
| Handoff | Через макеты и комментарии | Частично структурирован | Есть mapping и notes | Есть AI-ready handoff rules | Handoff проверяется до разработки |
| Governance | Отсутствует | Есть неформальные владельцы | Есть review отдельных изменений | Есть правила изменений и human review | Governance встроен в lifecycle системы |
| Validation | Отсутствует | Ручная проверка отдельных экранов | Есть checklist для ключевых слоев | Есть validation prompts и pass/fail формат | Validation встроена в process и release checks |
| AI Layer | Не управляется | Используется эпизодически | Используется вручную с осторожностью | Регламентирован rules, prompts и review | Встроен в процесс, governance и метрики |

## Level 0 — Interface Chaos

| Область | Состояние |
| --- | --- |
| Токены | Значения задаются вручную. Цвета, отступы, радиусы и типографика не имеют стабильных имен. |
| Компоненты | Компоненты копируются между макетами или создаются заново. Нет единого контракта вариантов и состояний. |
| Документация | Решения хранятся в личной памяти команды, комментариях или отдельных файлах. |
| Handoff | Разработчики интерпретируют макеты вручную. Требования к состояниям и edge cases часто теряются. |
| AI Layer | AI может генерировать интерфейсы без понимания системы, потому что нет правил, prompts и validation. |
| Основные риски | Быстро растет визуальная и техническая неконсистентность. AI усиливает хаос, потому что ему не на что опираться. |
| Переход дальше | Зафиксировать Foundation, базовые Design Tokens, список ключевых Components и минимальные правила Documentation. |

## Level 1 — Basic Design System

| Область | Состояние |
| --- | --- |
| Токены | Есть базовые значения для цвета, типографики и spacing, но они могут быть привязаны к визуальным названиям. |
| Компоненты | Есть основные компоненты, но не все варианты, состояния и правила использования описаны. |
| Документация | Документация помогает людям, но не всегда дает точный контекст для AI-ассистентов. |
| Handoff | Используются макеты, ссылки и комментарии. Mapping в код частичный. |
| AI Layer | AI применяется для ускорения текстов, идей и черновиков, но без устойчивых ограничений, source of truth и human review checklist. |
| Основные риски | AI может предлагать решения, которые выглядят системно, но обходят реальные компоненты и токены. |
| Переход дальше | Перейти к семантическим Design Tokens, описать component specs, states, props и UX Patterns. |

## Level 2 — Systematic Design System

| Область | Состояние |
| --- | --- |
| Токены | Есть иерархия primitive, semantic и component tokens. Использование токенов документировано. |
| Компоненты | Компоненты имеют варианты, состояния, anatomy, behavior и ограничения. |
| Документация | Документация покрывает назначение, usage, anti-patterns и handoff notes. |
| Handoff | Есть связь между design artifact и code artifact, но AI usage rules еще неполные. |
| AI Layer | AI может помогать в аудите, draft specs и подготовке acceptance criteria под контролем человека. |
| Основные риски | Без явных AI rules ассистент может ошибочно расширять систему или создавать неподдержанные варианты. |
| Переход дальше | Добавить AI usage rules, validation prompts, contract-based handoff и human review checklist. |

## Level 3 — AI-ready Design System

| Область | Состояние |
| --- | --- |
| Токены | Токены описаны как source of truth и пригодны для проверки AI-generated outputs. |
| Компоненты | Specs включают props, states, behavior, accessibility, code mapping и AI usage rules. |
| Документация | Документация структурирована для дизайнеров, разработчиков, аналитиков и AI-ассистентов. |
| Handoff | Handoff включает token mapping, component contract, edge cases и acceptance criteria. |
| AI Layer | AI работает внутри правил: audit, Validation, Handoff notes, risk analysis, draft patterns и explicit human review. |
| Основные риски | Риск смещается с хаотичной генерации на качество governance и полноту validation. |
| Переход дальше | Встроить AI-assisted validation в регулярный процесс design review, development review и release checks. |

## Level 4 — AI-assisted Interface Infrastructure

| Область | Состояние |
| --- | --- |
| Токены | Изменения токенов проходят impact analysis и автоматизированную проверку применения. |
| Компоненты | Компоненты связаны с кодом, документацией, тестами и AI-readable контрактами. |
| Документация | Документация обновляется как часть жизненного цикла системы и используется в validation pipeline. |
| Handoff | Handoff становится повторяемым процессом с проверкой до разработки и после реализации. |
| AI Layer | AI встроен в аудит, генерацию черновиков, проверку соответствия, подготовку Handoff и релизных материалов. |
| Основные риски | Необходим контроль качества AI-процессов, версионирование правил и ответственность владельцев системы. |
| Переход дальше | Поддерживать governance, метрики качества, обучение команды и регулярные аудиты зрелости. |

## Как использовать модель

- Определить текущий уровень по каждому слою системы.
- Зафиксировать самый слабый слой как основной риск.
- Сформировать roadmap перехода на следующий уровень.
- Не внедрять AI-assisted workflows раньше, чем появятся правила, токены, specs и validation.

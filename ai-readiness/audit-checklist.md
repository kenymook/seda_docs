# SEDA AI Readiness Audit Checklist

Этот чеклист помогает оценить готовность дизайн-системы к AI-assisted product development. Он проверяет не только наличие артефактов, но и способность системы быть source of truth для людей и AI-ассистентов.

## Как использовать

- Отмечайте пункт как выполненный только при наличии проверяемого артефакта.
- Если ответ зависит от устной договоренности, пункт считается невыполненным.
- Для каждого риска фиксируйте owner, impact и следующий шаг.

## Foundation

- [ ] Определены базовые принципы интерфейса?
  - Почему это важно: AI должен понимать, какие решения считаются допустимыми.
  - Хорошее состояние: принципы описаны в документации и применяются при review.
- [ ] Описана визуальная роль цвета, типографики, spacing и elevation?
  - Почему это важно: без foundation AI будет подбирать визуальные решения произвольно.
  - Хорошее состояние: foundation связан с token architecture.
- [ ] Есть правила композиции экранов?
  - Почему это важно: AI часто генерирует убедительные, но несистемные layout-решения.
  - Хорошее состояние: есть правила сетки, плотности, иерархии и responsive behavior.
- [ ] Описаны принципы accessibility?
  - Почему это важно: accessibility нельзя оставлять на позднюю ручную проверку.
  - Хорошее состояние: правила контраста, focus, keyboard и semantics встроены в specs.
- [ ] Есть единые принципы tone of voice для интерфейсных текстов?
  - Почему это важно: AI может менять тональность между экранами.
  - Хорошее состояние: microcopy rules доступны для design и product tasks.
- [ ] Описаны продуктовые ограничения для интерфейсов?
  - Почему это важно: AI не знает локальные бизнес-правила без явного контекста.
  - Хорошее состояние: ограничения связаны с UX Patterns и acceptance criteria.
- [ ] Есть правила для плотных рабочих интерфейсов?
  - Почему это важно: AI склонен добавлять декоративные элементы, если задача не ограничена.
  - Хорошее состояние: documented density, hierarchy и scanning rules.
- [ ] Foundation имеет владельца и процесс обновления?
  - Почему это важно: AI-ready система требует governance.
  - Хорошее состояние: изменения проходят review и версионирование.

## Design Tokens

- [ ] Есть разделение primitive, semantic и component tokens?
  - Почему это важно: AI должен выбирать смысловые токены, а не случайные значения.
  - Хорошее состояние: уровни токенов описаны и связаны между собой.
- [ ] Все ключевые визуальные решения выражены через токены?
  - Почему это важно: hardcoded values разрушают проверяемость.
  - Хорошее состояние: цвета, spacing, radius, typography и shadows используют токены.
- [ ] Названия токенов отражают назначение, а не внешний вид?
  - Почему это важно: semantic naming помогает переносить решения между темами и платформами.
  - Хорошее состояние: токены названы по роли в интерфейсе.
- [ ] Есть правила добавления новых токенов?
  - Почему это важно: AI и люди не должны расширять систему без governance.
  - Хорошее состояние: описан процесс запроса, review и публикации токена.
- [ ] Есть mapping токенов между design и code?
  - Почему это важно: handoff должен быть проверяемым.
  - Хорошее состояние: каждый используемый токен имеет code reference.
- [ ] Описаны deprecated tokens?
  - Почему это важно: AI может использовать устаревшие значения, если они остаются в документации.
  - Хорошее состояние: deprecated tokens помечены с заменой и сроком удаления.
- [ ] Есть правила платформенных отличий?
  - Почему это важно: web, iOS и Android могут требовать разных реализаций одного смысла.
  - Хорошее состояние: различия описаны на уровне mapping, а не через визуальные исключения.
- [ ] Есть Validation для Design Token usage?
  - Почему это важно: AI-generated output должен проверяться на соответствие архитектуре.
  - Хорошее состояние: есть prompts, checklist или автоматические проверки.

## Components

- [ ] У каждого ключевого компонента есть назначение?
  - Почему это важно: AI должен знать, когда компонент уместен.
  - Хорошее состояние: purpose описан в component spec.
- [ ] Описаны when to use и when not to use?
  - Почему это важно: это снижает риск неверного выбора компонента.
  - Хорошее состояние: есть конкретные сценарии и ограничения.
- [ ] Описана anatomy компонента?
  - Почему это важно: AI должен понимать составные части и их обязательность.
  - Хорошее состояние: anatomy связана с props и slots.
- [ ] Описаны все состояния?
  - Почему это важно: loading, error, disabled, focus и empty states часто теряются.
  - Хорошее состояние: states описаны визуально, поведенчески и технически.
- [ ] Описаны variants и их ограничения?
  - Почему это важно: AI может создавать неподдержанные комбинации.
  - Хорошее состояние: допустимые combinations перечислены.
- [ ] Есть props contract?
  - Почему это важно: design-to-code mapping требует точного контракта.
  - Хорошее состояние: props имеют типы, значения по умолчанию и ограничения.
- [ ] Компоненты связаны с токенами?
  - Почему это важно: визуальная реализация должна быть проверяемой.
  - Хорошее состояние: component tokens указаны в spec.
- [ ] Есть anti-patterns?
  - Почему это важно: AI часто предлагает формально похожие, но неверные решения.
  - Хорошее состояние: запрещенные паттерны описаны с причиной.

## UX Patterns

- [ ] Описаны ключевые пользовательские сценарии?
  - Почему это важно: AI должен работать с задачами, а не только с экранами.
  - Хорошее состояние: patterns привязаны к user goals.
- [ ] Есть правила для форм?
  - Почему это важно: формы требуют структуры, validation, errors и accessibility.
  - Хорошее состояние: описаны field groups, required fields, submit behavior и errors.
- [ ] Есть правила empty states?
  - Почему это важно: AI часто генерирует пустые состояния без полезного действия.
  - Хорошее состояние: empty states имеют причину, текст и next action.
- [ ] Есть правила error states?
  - Почему это важно: ошибки должны помогать восстановлению, а не только сообщать о проблеме.
  - Хорошее состояние: error states включают cause, recovery и severity.
- [ ] Есть правила loading states?
  - Почему это важно: loading влияет на восприятие скорости и стабильности.
  - Хорошее состояние: описаны skeleton, spinner и progressive loading.
- [ ] Есть правила table/list patterns?
  - Почему это важно: рабочие интерфейсы требуют плотного, сканируемого представления данных.
  - Хорошее состояние: описаны sorting, filtering, pagination и empty results.
- [ ] Есть правила navigation patterns?
  - Почему это важно: AI может нарушить информационную архитектуру.
  - Хорошее состояние: navigation rules связаны с масштабом продукта.
- [ ] Есть правила подтверждений и destructive actions?
  - Почему это важно: AI должен учитывать риск действий пользователя.
  - Хорошее состояние: описаны confirmations, undo и severity.

## Documentation

- [ ] Документация имеет единую структуру?
  - Почему это важно: AI лучше работает с повторяемыми шаблонами.
  - Хорошее состояние: component specs используют общий шаблон.
- [ ] Документация покрывает дизайнеров и разработчиков?
  - Почему это важно: разные роли используют разные уровни детализации.
  - Хорошее состояние: есть design usage, props, code mapping и handoff notes.
- [ ] Документация пригодна для аналитиков?
  - Почему это важно: acceptance criteria и продуктовые правила часто приходят от аналитиков.
  - Хорошее состояние: есть behavior, constraints и сценарии.
- [ ] Документация содержит AI usage rules?
  - Почему это важно: AI должен знать, что разрешено, а что требует review.
  - Хорошее состояние: правила встроены в specs и prompts.
- [ ] Есть examples и anti-patterns?
  - Почему это важно: примеры помогают отличать правильное применение от похожего неправильного.
  - Хорошее состояние: examples покрывают типичные и пограничные случаи.
- [ ] Документация версионируется?
  - Почему это важно: AI и команда должны понимать актуальность правил.
  - Хорошее состояние: есть changelog или release notes.
- [ ] У документации есть owner?
  - Почему это важно: source of truth требует ответственности.
  - Хорошее состояние: владелец указан для каждого раздела или слоя.
- [ ] Устаревшая документация помечается?
  - Почему это важно: AI может использовать старые правила, если они не отмечены.
  - Хорошее состояние: deprecated pages имеют replacement guidance.

## Handoff

- [ ] Есть mapping design artifact to code artifact?
  - Почему это важно: разработка должна получать не картинку, а контракт реализации.
  - Хорошее состояние: компоненты, props и tokens связаны с кодом.
- [ ] Описаны обязательные состояния для реализации?
  - Почему это важно: состояния часто пропускаются при передаче макета.
  - Хорошее состояние: state matrix включена в handoff.
- [ ] Есть правила responsive behavior?
  - Почему это важно: AI и разработчики должны понимать поведение layout на разных ширинах.
  - Хорошее состояние: breakpoints, wrapping и density описаны явно.
- [ ] Есть platform-specific notes?
  - Почему это важно: web, iOS и Android требуют разных технических решений.
  - Хорошее состояние: различия описаны в handoff notes.
- [ ] Есть acceptance criteria?
  - Почему это важно: без критериев сложно проверить итоговую реализацию.
  - Хорошее состояние: criteria покрывают visual, behavior, accessibility и edge cases.
- [ ] Handoff notes можно генерировать AI-assisted способом?
  - Почему это важно: AI может ускорить подготовку, если есть шаблон и правила.
  - Хорошее состояние: notes создаются по структуре и проходят review.
- [ ] Есть список human review checks?
  - Почему это важно: AI не должен быть финальным валидатором.
  - Хорошее состояние: checklist включен в процесс передачи.
- [ ] Есть traceability от требования до компонента?
  - Почему это важно: помогает понять, почему выбран конкретный UI pattern.
  - Хорошее состояние: requirement, pattern, component и code artifact связаны.

## Governance

- [ ] Определены владельцы Foundation, Design Tokens, Components и Documentation?
  - Почему это важно: AI-ready система требует управляемости.
  - Хорошее состояние: ownership закреплен и видим команде.
- [ ] Есть процесс добавления новых компонентов?
  - Почему это важно: AI не должен легитимизировать случайные элементы.
  - Хорошее состояние: есть request, review, approval и release.
- [ ] Есть процесс изменения токенов?
  - Почему это важно: изменения токенов влияют на множество интерфейсов.
  - Хорошее состояние: есть impact analysis и миграционный план.
- [ ] Есть правила использования AI в команде?
  - Почему это важно: разные участники могут применять AI с разным уровнем риска.
  - Хорошее состояние: правила описаны по типам задач и зонам ответственности.
- [ ] Есть review для AI-generated artifacts?
  - Почему это важно: AI outputs требуют проверки человеком.
  - Хорошее состояние: review включен в workflow.
- [ ] Есть правила хранения и обновления prompts?
  - Почему это важно: prompts становятся частью рабочей инфраструктуры.
  - Хорошее состояние: prompts версионируются и тестируются на примерах.
- [ ] Есть критерии выпуска изменений дизайн-системы?
  - Почему это важно: система должна развиваться предсказуемо.
  - Хорошее состояние: release criteria покрывают Documentation, Design Tokens, Components и Validation.
- [ ] Есть способ сообщать о несоответствиях?
  - Почему это важно: команды должны быстро фиксировать системные расхождения.
  - Хорошее состояние: есть intake process для issues и proposals.

## Validation

- [ ] Есть visual consistency validation?
  - Почему это важно: AI-generated UI может выглядеть близко, но отличаться в деталях.
  - Хорошее состояние: проверяются spacing, hierarchy, typography, density и states.
- [ ] Есть token validation?
  - Почему это важно: hardcoded values подрывают source of truth.
  - Хорошее состояние: проверяется применение semantic и component tokens.
- [ ] Есть Component usage validation?
  - Почему это важно: неверный компонент может нарушить сценарий.
  - Хорошее состояние: проверяется соответствие purpose, variants и states.
- [ ] Есть accessibility validation?
  - Почему это важно: AI не гарантирует доступность по умолчанию.
  - Хорошее состояние: проверяются contrast, focus, keyboard, labels и semantics.
- [ ] Есть handoff validation?
  - Почему это важно: неполный handoff создает ошибки в реализации.
  - Хорошее состояние: проверяются mapping, props, tokens и acceptance criteria.
- [ ] Есть AI output validation?
  - Почему это важно: AI outputs должны оцениваться системно.
  - Хорошее состояние: есть формат отчета с risks, violations и recommendations.
- [ ] Есть проверка edge cases?
  - Почему это важно: AI часто показывает happy path.
  - Хорошее состояние: проверяются empty, error, loading, permissions и long content.
- [ ] Есть критерии pass/fail?
  - Почему это важно: validation должна быть однозначной.
  - Хорошее состояние: результат проверки содержит статус, severity и next action.

## AI Readiness

Этот раздел проверяет AI Layer: правила, prompts, validation, ограничения и human review для AI-assisted product development.

- [ ] Описаны разрешенные AI use cases?
  - Почему это важно: не все задачи одинаково безопасны для AI.
  - Хорошее состояние: use cases разделены на allowed, assisted и restricted.
- [ ] Описаны запрещенные AI use cases?
  - Почему это важно: система должна предотвращать рискованные применения.
  - Хорошее состояние: запреты включают генерацию неподдержанных компонентов и обход правил.
- [ ] Есть prompt library для системных задач?
  - Почему это важно: prompts должны работать внутри дизайн-системы.
  - Хорошее состояние: prompts покрывают audit, Design Tokens, Components, Handoff и Validation.
- [ ] Есть правила предоставления контекста AI?
  - Почему это важно: качество ответа зависит от качества входных данных.
  - Хорошее состояние: prompts требуют specs, tokens, screen context и constraints.
- [ ] Есть human validation для AI outputs?
  - Почему это важно: AI ускоряет, но не принимает финальное решение.
  - Хорошее состояние: каждый AI output имеет owner и review step.
- [ ] Есть risk classification?
  - Почему это важно: разные ошибки имеют разную цену.
  - Хорошее состояние: risks классифицируются по severity и impact.
- [ ] Есть метрики AI-assisted качества?
  - Почему это важно: нужно понимать, ускорение не ухудшает систему.
  - Хорошее состояние: отслеживаются consistency issues, review time и rework.
- [ ] Есть план развития AI-readiness?
  - Почему это важно: готовность к AI является процессом, а не разовой настройкой.
  - Хорошее состояние: roadmap связан с maturity model и audit findings.
